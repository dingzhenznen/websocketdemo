  - WebSocket 连接的是你的 Node 服务

  - shell 进程挂在 slave 上

  所以链路是：

  浏览器 xterm
    <-> WebSocket
  Node server
    <-> PTY master
  PTY slave
    <-> shell(bash/zsh)

  不是“ws 直接连 master、shell 直接连 slave”，而是：

  - ws 连的是 server.js
  - server.js 再用 term.write() / term.onData() 操作 PTY 的 master 侧
  - shell 则把 slave 当作自己的 stdin/stdout/stderr 所在终端

  - shell 输出 -> 写到 slave -> PTY master 收到 -> term.onData(...) -> WebSocket -> 前端显示

• 可以用这张图理解：

  [ Browser / xterm.js ]
          |
          | WebSocket
          v
  [ server.js ]
          |
          | term.write() / term.onData()
          v
  [ PTY master ] <==== PTY pair ====> [ PTY slave ]
                                           |
                                           | stdin / stdout / stderr
                                           v
                                     [ bash / zsh ]

  对应到你的 websocketdemo/node-pty/server.js：

  const term = pty.spawn(shell, [], {
    name: 'xterm-256color',
    cols: 80,
    rows: 24,
    cwd: process.cwd(),
    env: process.env,
  });

  这里 pty.spawn(...) 做了两件事：

  1. 创建一个 PTY 对
  2. 启动一个 shell，让 shell 绑定到 slave 端

  而返回给 JS 的 term，本质上就是你在操作 master 端的接口。

  所以：

  - term.write(data)：等价于往 master 写数据，shell 会像用户敲键盘一样从 slave 收到
  - term.onData(...)：等价于监听 master 读到的数据，这些数据来自 shell 写到 slave 的输出
  - term.resize(cols, rows)：修改这个终端设备的窗口大小，shell 那边会感知到

  再拆成两条流：

  1. 输入流

  前端键盘输入
  -> WebSocket 发到 server.js
  -> term.write(...)
  -> PTY master
  -> PTY slave
  -> shell 读取 stdin
  -> shell 执行命令

  2. 输出流

  shell 输出 stdout/stderr
  -> PTY slave
  -> PTY master
  -> term.onData(...)
  -> WebSocket 发回前端
  -> xterm.js 渲染

  为什么 shell 必须挂在 slave，不是直接挂 master：

  - 对 shell 来说，slave 看起来就是一个标准终端设备 /dev/tty
  - shell 和很多 TUI 程序（vim、top、less）都依赖这种“我连着一个真实终端”的语义
  - master 是给外部控制程序用的，不是给应用进程直接当终端设备用的

  所以最准确的一句话是：

  Node 通过 node-pty 控制 PTY 的 master 端，shell 运行在 slave 端，WebSocket 只是把浏览器和 master 端桥接起来。
