现在这套交互，本质上是一个“浏览器里的 xterm.js 终端”通过 WebSocket 连接到 Node 服务端，服务端再把输入输出桥接到一个本地 PTY
  shell。

  整体链路

  1. 浏览器打开页面 /，拿到 websocketdemo/node-pty/public/index.html。
  2. 页面里初始化 xterm.js 终端，并连接 ws://host/ws。
  3. 服务端 websocketdemo/node-pty/server.js 收到 WebSocket 连接后，立刻 pty.spawn(...) 启一个真实 shell。
  4. 前端输入的每个按键，会通过 WebSocket 发给服务端。
  5. 服务端把这些输入写进 PTY。
  6. PTY 产生的输出，再通过 WebSocket 回推给前端，前端写进终端界面。
  7. 浏览器窗口变化时，前端把新的 cols/rows 发给服务端，服务端同步调整 PTY 大小。
  8. WebSocket 断开时，服务端杀掉这个 PTY 会话。

  按代码拆开看

  1. 服务启动
  在 websocketdemo/node-pty/server.js 里：

  - express 提供静态页面和前端依赖
  - ws 提供 WebSocket 服务，路径固定是 /ws
  - node-pty 负责创建伪终端
  - ensureSpawnHelperExecutable() 会在 macOS 下给 node-pty 的 spawn-helper 补执行权限，代码在 websocketdemo/node-pty/pty-
    helper.js

  这一步只是把运行环境准备好。

  2. 前端页面初始化
  在 websocketdemo/node-pty/public/index.html：

  - new Terminal(...) 创建浏览器终端组件
  - FitAddon 用来根据容器尺寸自动计算终端列数和行数
  - term.open(...) 把终端挂到页面
  - fitAddon.fit() 先做一次初始尺寸计算

  然后创建 WebSocket：

  const ws = new WebSocket(`${protocol}://${location.host}/ws`);

  也就是说，页面和服务端之间的唯一实时通信通道就是这个 /ws。

  3. WebSocket 建连后，服务端创建 PTY
  在 websocketdemo/node-pty/server.js 的 wss.on('connection', ...) 里：

  const term = pty.spawn(shell, [], {
    name: 'xterm-256color',
    cols: 80,
    rows: 24,
    cwd: process.cwd(),
    env: process.env,
  });

  这里每来一个 WebSocket 连接，就新建一个独立的 shell 会话。

  关键点：

  - shell 是 process.env.SHELL 或 bash
  - cwd 是启动 server.js 时的当前目录
  - env 直接继承当前 Node 进程环境变量
  - 初始大小先给 80x24，后面会被前端 resize 覆盖

  所以：一个浏览器连接，对应一个独立 PTY shell。

  4. 服务端先发 ready，前端显示连接信息
  PTY 创建后，服务端先发：

  ws.send(JSON.stringify({ type: 'ready', shell }));

  前端收到后：

  if (message.type === 'ready') {
    term.writeln(`Connected to ${message.shell}`);
  }

  这只是一个“会话已准备好”的提示，不是真正 shell 输出。

  5. 前端输入如何传到 shell
  前端这段最关键：

  term.onData((data) => {
    ws.send(JSON.stringify({ type: 'input', data }));
  });

  意思是：

  - 用户在 xterm 里敲键盘
  - xterm 产生字符流 data
  - 前端把它包装成 { type: 'input', data }
  - 通过 WebSocket 发给服务端

  服务端收到消息后：

  if (message.type === 'input' && typeof message.data === 'string') {
    term.write(message.data);
    return;
  }

  这里的 term 是 node-pty 的 PTY 实例，不是前端 xterm。
  term.write(message.data) 就是把用户输入真正写进 shell。

  所以输入路径是：

  浏览器键盘 -> xterm.onData -> ws.send(input) -> server ws.on(message) -> pty.write()

  6. shell 输出如何回到前端
  服务端监听 PTY 输出：

  term.onData((data) => {
    if (ws.readyState === WebSocket.OPEN) {
      ws.send(JSON.stringify({ type: 'output', data }));
    }
  });

  前端收到后：

  if (message.type === 'output') {
    term.write(message.data);
    return;
  }

  所以输出路径是：

  PTY shell stdout/stderr -> pty.onData -> ws.send(output) -> 浏览器 ws.onmessage -> xterm.write()

  注意这里没有区分 stdout 和 stderr，所有终端输出都统一按 output 回给前端。这很符合终端场景，因为 PTY 本身就是字符流语义，不像
  普通 spawn 会分 stdout/stderr 两条管道。

  7. resize 交互
  前端在两个时机发 resize：

  1. WebSocket 刚连接成功时
  2. 浏览器窗口尺寸变化时

  代码在 websocketdemo/node-pty/public/index.html：

  ws.send(JSON.stringify({
    type: 'resize',
    cols: term.cols,
    rows: term.rows
  }));

  服务端收到后：

  if (
    message.type === 'resize' &&
    Number.isInteger(message.cols) &&
    Number.isInteger(message.rows) &&
    message.cols > 0 &&
    message.rows > 0
  ) {
    term.resize(message.cols, message.rows);
  }

  这一步很重要，因为 shell 里很多程序依赖终端尺寸，比如：

  - vim
  - less
  - top
  - 命令换行布局

  所以 resize 路径是：

  浏览器容器变化 -> fitAddon 重新算 cols/rows -> ws.send(resize) -> pty.resize()

  8. 退出和断开
  如果 shell 退出，服务端会发：

  ws.send(JSON.stringify({ type: 'exit', exitCode, signal }));
  ws.close();

  前端收到：

  term.writeln(`\r\n[process exited code=${message.exitCode}]`);
  statusEl.textContent = 'closed';

  如果浏览器先断开连接，服务端会：

  ws.on('close', () => {
    term.kill();
  });

  也就是：

  - 前端关页面/断网/WebSocket 关闭
  - 对应的 PTY 会被杀掉
  - 不会残留后台 shell 进程

  当前协议很简单，只有 4 类消息

  前端 -> 服务端：

  - input: 用户输入字符
  - resize: 终端大小变化

  服务端 -> 前端：

  - ready: shell 已创建
  - output: shell 输出
  - exit: shell 退出

  这套逻辑的几个特点

  1. WebSocket 连接 = 一个 PTY 会话
  2. 服务端不做命令解析，只做字节流/字符流转发
  3. 前端不维护命令状态，只负责显示和转发输入
  4. 这是“真终端”模型，不是“发一条命令，回一段结果”模型
  5. 会话是临时的，连接断开后 PTY 直接销毁

› 客户端的输入 会通过 pty 在shell 中执行是吗


• 是。

  更准确地说，客户端输入的内容会被当成“终端键盘输入”写进 PTY，再由 PTY 后面的 shell 来解释和执行，不是服务端自己解析命令。

  对应代码在 websocketdemo/node-pty/server.js：

  - 前端发消息：

  { type: 'input', data }

  - 服务端收到后：

  if (message.type === 'input' && typeof message.data === 'string') {
    term.write(message.data);
    return;
  }

  这里的 term.write(...) 是把字符写进伪终端。比如你在前端输入：

  ls

  再按回车，实际过程是：

  1. 前端把 l、s、\r 这类输入发给服务端
  2. 服务端 term.write(...) 写入 PTY
  3. shell 收到这串输入
  4. shell 在遇到回车后把它当一条命令执行
  5. 执行结果通过 term.onData(...) 回传前端

  所以关键区别是：

  - 输入字符本身不是“立刻执行”
  - 是 shell 收到完整终端输入后，按自己的规则执行
  - 回车、Ctrl+C、方向键、Tab，这些也都会作为终端输入传进去

  这也是为什么它像一个真实终端，而不是一个“远程 RPC 调命令”的接口。
